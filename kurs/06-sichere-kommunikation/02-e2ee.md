# Ende-zu-Ende-Verschlüsselung: Warum TLS manchmal nicht genug ist

TLS verschlüsselt die Verbindung zwischen deinem Gerät und dem Server. Das ist gut und notwendig. Aber es hat eine grundlegende Grenze.

## Das Problem mit TLS allein

TLS verschlüsselt den *Weg*. Nicht die *Nachricht*.

```
Alice                  Server                Bob
  │                      │                    │
  │──── TLS ────────────►│                    │
  │    "Hallo Bob"       │──── TLS ──────────►│
  │                      │    "Hallo Bob"     │
  │                      │                    │
  │                Der Server sieht           │
  │           die Nachricht im Klartext       │
```

Der Server muss die Nachricht entschlüsseln, um zu wissen, an wen er sie weiterleiten soll. Das ist keine Schwäche der Implementierung. Es ist strukturell so, wie eine server-basierte Kommunikationsarchitektur funktioniert.

**Konsequenz:** Wer den Server kontrolliert, kann alle Nachrichten lesen. Das gilt für den Betreiber selbst, für Angreifer die den Server kompromittieren, und für Behörden die mit einer Anfrage ankommen.

## Die Idee hinter E2EE

Ende-zu-Ende-Verschlüsselung löst dieses Problem, indem der Schlüssel zum Entschlüsseln nie den Server erreicht. Die Nachricht wird auf Alices Gerät verschlüsselt und erst auf Bobs Gerät entschlüsselt. Der Server transportiert nur Chiffrat. Er kann die eigentliche Nachricht nicht lesen.

```
Alice                  Server                   Bob
  │                      │                       │
  │── verschlüsselt ────►│── verschlüsselt ─────►│
  │  [x7$#@kLm9...]      │  [x7$#@kLm9...]       │
  │                      │                       │
  │               Der Server sieht               │
  │           nur unlesbares Chiffrat            │
```

## Das eigentliche Problem: Schlüsselaustausch

Damit Alice eine Nachricht verschlüsseln kann, die nur Bob lesen kann, braucht sie Bobs Schlüssel. Aber wie tauscht man Schlüssel sicher aus, ohne sich vorher persönlich getroffen zu haben?

Das ist das Kernproblem der asymmetrischen Kryptographie und es ist seit Jahrzehnten gelöst. Die Lösung kann aber Konsequenzen für die Nutzbarkeit haben

### Asymmetrische Schlüssel (kurz)

Jede Person hat zwei Schlüssel:
- **Public Key**: Kann frei verteilt werden. Damit verschlüsseln andere Nachrichten an dich.
- **Private Key**: Bleibt geheim auf deinem Gerät. Damit entschlüsselst du.

Was mit dem Public Key verschlüsselt wurde, kann nur der Private Key entschlüsseln und umgekehrt. Der Server speichert Public Keys, aber nie Private Keys. Das ist die Grundlage von E2EE.

Konkrete Algorithmen die dabei zum Einsatz kommen: **Diffie-Hellman** für den Schlüsselaustausch, **AES** für die eigentliche Verschlüsselung, **Double Ratchet** (Signal-Protokoll) für sichere Einzel-Chats, **Megolm** (Matrix) für Gruppen-Chats.

## Was E2EE nicht löst: Das Verifikationsproblem

Wenn Bob dir seinen Public Key schickt, woher weißt du, dass es wirklich Bobs Key ist und nicht der eines Angreifers, der sich dazwischengeschaltet hat?

Das nennt sich **Man-in-the-Middle-Angriff**, und E2EE verhindert ihn nicht automatisch. Verhindert wird er durch **Verifikation**: Du vergleichst Bobs Schlüssel-Fingerprint persönlich oder über einen unabhängigen Kanal.

In der Praxis macht das kaum jemand. Das ist eine bewusste Entscheidung vieler Nutzer: Sie vertrauen darauf, dass der Server keinen gefälschten Key liefert. Wer einem sehr gut organisierten Angreifer gegenübersteht (Staatstrojaner, gezielte Überwachung), sollte Schlüssel verifizieren.

## Die Kompromisse von E2EE

E2EE ist kein Allheilmittel. Es bringt reale Nachteile:

- **Schlüsselverwaltung wird zum Nutzerproblem.** Wenn du dein Gerät verlierst oder zurücksetzt, sind verschlüsselte Nachrichten ohne Key-Backup unwiederbringlich verloren. TLS-basierte Dienste können dein Konto einfach auf einem neuen Gerät wiederherstellen. Bei E2EE ist das strukturell nicht so einfach möglich.
- **Mehrere Geräte sind komplizierter.** Jedes Gerät hat eigene Schlüssel. Damit alle deine Geräte Nachrichten lesen können, müssen Schlüssel zwischen Geräten synchronisiert werden, was wiederum Sicherheitsfragen aufwirft.
- **Serverseitige Suche und Backup entfallen.** Gmail kann deine Mails durchsuchen, weil es sie lesen kann. Ein E2EE-Dienst kann das nicht.
- **Metadaten bleiben sichtbar.** E2EE schützt den Inhalt. Aber der Server sieht weiterhin: Wer kommuniziert mit wem, wann, wie oft, und wie groß die Nachrichten sind. Das ist oft genug Information.

## Zusammenfassung

|                                       | TLS  | E2EE |
|---------------------------------------|------|------|
| Schützt vor dem Netzwerk              | ja   | ja   |
| Schützt vor dem Betreiber             | nein | ja   |
| Funktioniert ohne Schlüsselverwaltung | ja   | nein |
| Einfache Gerätewiederherstellung      | ja   | nein |
| Serverseitige Suche möglich           | ja   | nein |
| Metadaten geschützt                   | nein | nein |

E2EE löst ein spezifisches Problem: Vertrauen in den Betreiber wird optional. Alles andere bleibt eine Abwägung.
